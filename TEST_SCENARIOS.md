## CI Reconciliation Configuration

Each CI workflow runs a single scenario per product:

| Workflow | Product | Scenario | Command |
|----------|---------|----------|---------|
| `generate-bst.yml` | BST | basic | `--package bst --scenario basic` |
| `generate-se.yml` | SE | advanced | `--package se --scenario advanced` |
| `generate-ltd.yml` | Ltd | full | `--package ltd --scenario full` |
| `generate-taxi.yml` | Taxi | basic | `--package taxi --scenario basic` |

The `test.yml` workflow runs `node scripts/extract-scenarios.cjs` followed by `git diff --exit-code` to verify fixtures stay in sync with master data.

## CELL_MAP Pattern

Each product module (`app/products/*.js`) uses a `CELL_MAP` array as the single source of truth for:
- Which cells to read from the recalculated spreadsheet
- English labels for the reconciliation report
- diya-gl schema property mappings
- XBRL taxonomy concept references
- Report section grouping

Functions `standardReads()`, `reportSections()`, and `cellLabels()` all derive from `CELL_MAP`. This ensures E2E tests, reconciliation checks, and reports are always consistent.

## Multi-File Pipeline Extension

For SE and Ltd (multi-file products), the `runMultiFileSpreadsheet` function supports:
- `options.postHubRecalc` — array of filenames to recalculate AFTER the hub (e.g. `["Vat.xlsx"]` which reads from the hub)
- `options.additionalReads` — reads from recalculated leaf files (e.g. `{ "Bank.xlsx": { "Mar": ["A2"] } }`)

This enables reading VAT quarterly returns from Vat.xlsx and bank closing balances from Bank.xlsx/Cash.xlsx without changing the core pipeline.
